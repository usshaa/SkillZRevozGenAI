Tokenizer — Text is split into subword units called tokens. One word can be 1 token or many. Images become patches. Audio becomes mel-spectral frames.
Input type
Token IDs — Every token maps to a fixed integer in the vocabulary (50k–100k entries). IDs are not random — they are permanently assigned during vocabulary construction (BPE or WordPiece) and never change after training.
Vocabulary lookup
The same word always gets the same ID. "neural" = id 14925 forever in GPT-2 vocab. Context changes meaning, not ID.
Embedding vectors — Each token ID maps to a high-dimensional float vector (~768–4096 dims) via a learned lookup table. Vectors encode meaning — similar words cluster nearby in this space. Weights are learned via backprop during pretraining.

Transformer attention — For each query token, the model computes a dot-product score against every other token (Q·K), applies softmax, and uses these weights to sum up values (V). High weights = "pays attention to". This is how meaning is resolved from context.

Q·K·V mechanism
Vector database — Knowledge is stored as high-dim embedding vectors (from documents, code, images, etc). At query time, the query embedding is compared to all stored vectors using cosine similarity or FAISS ANN. The closest vectors = most relevant content.

Retrieval (RAG pipeline) — The query is embedded → nearest chunks are fetched from vector DB → scores are ranked → top chunks are injected into the prompt context window → the transformer now "knows" this information for generation.
Autoregressive generation — The transformer predicts the next token by attending over the full context (query + retrieved chunks). It samples from the probability distribution. Each new token is appended and the process repeats until EOS.
